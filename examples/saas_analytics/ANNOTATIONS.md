# SaaS Analytics ŌĆö Annotations

This document explains the provenance decisions and design choices in
`semantic.yaml` and `corrections.yaml` for the `saas_analytics` example. It
is intended as a human-readable companion to the machine-readable files and as
a guide for contributors extending this example.

---

## Why this example exists

Section 9 of `SPEC.md` lists `examples/saas_analytics/` as a canonical
example demonstrating patterns specific to SaaS product analytics:

- **Wide event tables** with an `event_type` discriminator
- **Semi-additive metrics** (DAU, MRR) that cannot be naively summed across time
- **Model-level `required_filters`** ensuring the events table is always queried
  with an `event_type` condition
- **Timestamp-heavy schemas** where multiple timestamp columns serve different
  analytical purposes

These patterns are distinct from the `financial_risk` example (which showcases
multi-clause joins, limit breach logic, and soft-delete patterns) and the
`ecommerce` example (which focuses on order funnels and inventory).

## Schema design decisions

### Events as a wide table

The `events` model uses a single physical table for all event types,
discriminated by the `event_type` column. This is a common SaaS analytics
pattern (also called an "activity schema"). The semantic.yaml models it with:

1. `model-level required_filters` ŌĆö `events.event_type IS NOT NULL` guards
   against data quality errors. The `reason` field explains why this matters.
2. `metric-level required_filters` ŌĆö each metric derived from events specifies
   the exact `event_type` value it requires (e.g., `event_type = 'user_active'`
   for DAU, `event_type = 'feature_used'` for feature_event_count).
3. `valid_values` on `event_type` ŌĆö provides the complete list of allowed values
   so agents know what can be safely used in filter conditions.
4. `difficulty: domain_dependent` on `event_type` ŌĆö signals that choosing the
   correct event_type requires business knowledge, not just schema inspection.

### Semi-additive metrics

Both `daily_active_users` and `monthly_recurring_revenue` use
`additivity: semi_additive`. This signals to consuming tools that:

- The metric **can** be disaggregated by user attributes (country, plan tier)
- The metric **cannot** be summed across the time dimension without explicit
  user instruction
- The `default_time_dimension` tells agents which column to use for time-series
  grouping

DAU uses `COUNT(DISTINCT ...)` with `aggregation: count_distinct`. MRR uses
`SUM` with `aggregation: sum`. Both carry `required_filters` that enforce the
conditions without which the metric is incorrect.

### Timestamp columns (role: timestamp)

The schema uses `role: timestamp` on six columns across five models:

| Model           | Column        | Purpose                                    |
|-----------------|---------------|--------------------------------------------|
| `users`         | `created_at`  | Signup cohort analysis                     |
| `users`         | `trial_ends_at` | Trial conversion analysis               |
| `users`         | `churned_at`  | Churn date filtering                       |
| `events`        | `occurred_at` | Primary event time axis (DAU, WAU, MAU)    |
| `sessions`      | `started_at`  | Session trend reporting                    |
| `sessions`      | `ended_at`    | Session duration (null = in progress)      |
| `subscriptions` | `started_at`  | Subscription cohort analysis, MRR trend    |
| `subscriptions` | `ended_at`    | Active subscription filtering              |

Having multiple timestamp columns is intentional and realistic. The
`known_gotchas` in the domain block explains when each should be used. The
`temporal_model.business_date_column` is set to `occurred_at` as the primary
time axis for event-based analysis.

### sessions.ended_at role: timestamp vs flag

`sessions.ended_at` is `role: timestamp` rather than `role: flag`. It carries
both a null-check semantic (in-progress sessions) and a time value when
non-null. `role: timestamp` is more accurate because consuming tools will use
this column in time arithmetic (duration calculation), not merely as a boolean
predicate.

## Provenance decisions

### Domain block: `llm_inferred` at confidence 0.83

The domain description was generated by the LLM pass and then lightly edited.
The key_concepts and known_gotchas were partially human-authored (the DAU
definition and the MRR semi-additivity caveat were added by the analytics team).
The domain block as a whole is `llm_inferred` because the majority of the text
originated from the LLM, but the confidence is elevated to 0.83 reflecting the
editorial review.

### Model provenances: `schema_constraint` at confidence 1.0

All five model blocks have `source_type: schema_constraint, confidence: 1.0`.
This indicates the physical table structure (columns, data types, primary keys,
foreign keys) was read directly from the database catalog ŌĆö there is no ambiguity
about what tables exist or their column types.

### Column provenances

Columns fall into three categories:

- **`schema_constraint, confidence: 1.0`** ŌĆö columns where the semantic meaning
  is unambiguous from the schema (primary keys, foreign keys, `NOT NULL`
  constraints). Examples: `user_id`, `event_id`, `subscription_id`.

- **`ddl_comment, confidence: 0.95`** ŌĆö columns with a DDL comment that was
  parsed directly. Example: `events.event_type` (the DDL comment listed the
  valid event types and the requirement to always filter).

- **`llm_inferred, confidence: 0.69ŌĆō0.83`** ŌĆö columns where the semantic
  meaning required interpretation. Examples: `subscriptions.mrr_usd` (the
  semi-additive nature is not obvious from the schema), `subscriptions.ended_at`
  (the grace-period interaction with `status` is a business rule, not a
  schema constraint).

### Concepts: `user_defined` vs `log_inferred`

- `active_user` and `monthly_recurring_revenue` are `user_defined` because they
  were written directly in `strata.md` by the analytics team. Their definition
  SQL matches the production dashboard queries exactly.
- `churn` is `log_inferred` because the definition SQL was reverse-engineered
  from query logs. The churn rate calculation (paid-plan filter, joining to plans
  to exclude free tier) was not documented in `strata.md` and was discovered
  from recurring query patterns.

## Corrections rationale

### corr_001 ŌĆö mrr_usd description override

The most common LLM error on MRR tables is treating MRR as a fully-additive
flow metric rather than a semi-additive stock metric. Agents were computing
"MRR" by summing `mrr_usd` over a date range, which produces cumulative contract
value (a different and larger number). The corrected description adds the
cross-time caveat and the free-plan clarification. This correction is approved
and in production.

### corr_002 ŌĆö ended_at description override

The inferred description suggested `ended_at IS NULL` as the active-subscription
filter. This is technically correct most of the time but breaks during dunning
grace periods. The corrected description explains the interaction between
`ended_at` and `status` and recommends `status = 'active'` as the safer filter.
This correction was validated by comparing MRR figures against the billing
system's own MRR calculation.

### corr_003 ŌĆö users.email suppression

`users.email` is PII and must not be exposed to LLM agents. The suppression
is permanent. Any query requiring email data must use the user management API.

### corr_004 ŌĆö active_user confidence override

The `active_user` concept was human-authored with a manually validated
definition. The LLM assigned confidence 0.84 based on the overall document
quality, but the concept itself is machine-checked against the production
dashboard. Overriding to 1.0 ensures consuming tools treat this definition as
authoritative.

### corr_005 ŌĆö avg_session_duration phantom session filter (pending)

An agent proposed filtering `duration_seconds > 5` to exclude phantom sessions
from load balancer health checks. The filter matches dashboard behaviour and
appears correct, but has not been reviewed by the engineering team who owns
the session computation logic. Marked `status: pending` until reviewed.

## What this example demonstrates that others do not

| Pattern | Where in this example |
|---|---|
| Wide event table with `event_type` discriminator | `events` model |
| Model-level `required_filters` for data quality | `events.event_type IS NOT NULL` |
| Metric-level `required_filters` for event type scoping | `daily_active_users`, `feature_event_count` |
| `count_distinct` aggregation with `semi_additive` | `daily_active_users` metric |
| Multiple `role: timestamp` columns with different analytical roles | `users`, `events`, `sessions`, `subscriptions` |
| `semi_additive` MRR metric with `default_time_dimension` | `monthly_recurring_revenue` metric |
| `valid_values` on a discriminator column | `events.event_type` |
| Grace-period interaction between two active-state columns | `subscriptions.ended_at` + `subscriptions.status` |
